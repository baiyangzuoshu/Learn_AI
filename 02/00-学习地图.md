# 学习地图：先分清这些概念

## 1. 一句话模型

> Model 负责推理和选择，Agent 负责循环与状态，Skill 负责教它怎样做，Tool 负责确定性执行，MCP 负责标准化连接，Eval 负责证明它做得是否可靠。

## 2. 概念边界

| 概念 | 核心职责 | 典型产物 | 不应该承担的职责 |
|---|---|---|---|
| Prompt | 描述当前一次任务 | 用户消息、系统指令 | 长期维护复杂团队流程 |
| `AGENTS.md` | 仓库内持久规则与约定 | 构建命令、编码规范、验证要求 | 大量教程、脚本或外部系统连接 |
| Tool | 接受明确输入并产生可验证输出 | CLI、函数、API | 自己决定高层任务目标 |
| Skill | 封装可复用的步骤、知识、脚本路由 | `SKILL.md`、references、scripts | 代替确定性业务实现或网络协议 |
| MCP | 让 Host 发现并调用外部能力或上下文 | Server、Tools、Resources、Prompts | 自动形成可靠 Agent 或替代业务权限设计 |
| Agent | 围绕目标反复观察、决策、调用工具并验证 | 运行循环、状态、审批、报告 | 无限制自主执行或自行证明正确 |
| Eval | 用场景、断言和证据衡量行为 | 测试集、评分器、运行记录 | 只凭主观印象给系统打分 |

## 3. 它们怎样组合

以“检查一个游戏 UI Prefab”为例：

1. 用户把目标交给 Agent。
2. Agent 根据 Skill 得知应先读哪些项目事实、调用什么检查器、完成后怎样报告。
3. Agent 通过本地 CLI 或 MCP Tool 调用确定性检查器。
4. MCP Server 校验输入、限制可访问目录、执行检查器并返回结构化结果。
5. Agent 根据结果决定是否继续读取 Resource、请求人工授权或结束。
6. Eval 检查 Agent 是否用了正确工具、是否越权、是否给出了可复核证据。

关键点：Skill 和 MCP 是互补关系。Skill 描述“工作流”，MCP 提供“外部能力和数据”。

## 4. Agent 的最小工作循环

先掌握单 Agent，不急着多 Agent：

```text
接收目标
  ↓
获取事实 / 观察环境
  ↓
形成可执行计划
  ↓
检查权限与风险
  ↓
调用工具执行
  ↓
验证结果
  ↓
报告证据，或带着新事实继续循环
```

实现时至少要回答：

- 状态保存在什么地方？
- 工具失败后是重试、降级还是停止？
- 哪些动作需要人工批准？
- 怎样避免把模型输出误当作事实？
- 最终结果由什么证据支撑？

## 5. MCP 的最小心智模型

- **Host**：承载 Agent 的应用，例如 Codex。
- **Client**：Host 内部与某个 MCP Server 维持连接的协议客户端。
- **Server**：对外声明并实现能力。
- **Tool**：模型可以选择调用的动作，有输入 Schema。
- **Resource**：可读取的上下文数据，以 URI 标识。
- **Prompt**：由用户或客户端选择的可复用提示模板。
- **Transport**：消息怎样传输；本地入门优先使用 `stdio`。
- **JSON-RPC**：请求、响应和通知的消息基础。

学习 MCP 时要分两层：

1. **协议层**：初始化、能力协商、`tools/list`、`tools/call`、错误和生命周期。
2. **业务层**：你的工具真正做什么、能访问哪里、错误怎样表达、如何测试。

`01` 中 `linkup-check` 是业务能力，`linkup-dev-mcp` 是协议适配层。不能把二者混成一份代码。

## 6. 前置知识自测

你已经有编程经验。以下项目用于确认是否需要做第 0 周的 Node/TypeScript 迁移练习；全部掌握即可直接进入第 1 周：

- 能使用 `cd`、`pwd`、`ls`、`rg` 和管道查看文件。
- 能阅读 `package.json`，理解 dependencies 和 scripts。
- 能运行 Node.js 脚本并看懂退出码。
- 能读写 JSON，并解释 JSON Schema 的 `type`、`properties`、`required`。
- 能用 JavaScript/TypeScript 写异步函数和处理异常。
- 能写至少一个自动化测试并区分 stdout/stderr。
- 能用 Git 查看 diff 和 status，知道不要覆盖无关改动。

不要求先学：模型训练数学、微调、向量数据库、RAG、多 Agent 框架或 Kubernetes。

## 7. 常见误区

1. **“Agent 就是一个很长的 Prompt”**：Prompt 只是输入；Agent 还需要循环、工具、状态、权限和终止条件。
2. **“写了 SKILL.md 就拥有新能力”**：Skill 能教流程，但底层工具不存在时，它不能凭空完成确定性动作。
3. **“MCP 是另一个 AI 模型”**：MCP 是连接协议，Server 可以完全不调用模型。
4. **“Tool 越多越强”**：工具面越大，误调用、权限和提示注入风险越高。优先最小能力。
5. **“一次成功演示就是完成”**：Agent 行为有不确定性，必须有重复场景、失败样例和证据。
6. **“先做多 Agent 更先进”**：如果一个 Agent 的工具契约和验收都不稳定，多 Agent 只会放大问题。

## 8. 路线分支

主线使用 Codex 作为现成 Agent Host，重点学习工程分层。完成第 7 周后再选分支：

- **应用开发分支**：使用 OpenAI Agents SDK，自行管理模型、工具执行、状态、审批和追踪。
- **Codex 工程分支**：继续强化仓库 Skill、MCP、hooks、权限和自动化。
- **协议分支**：深入 MCP transport、远程认证、通知、采样和跨客户端兼容性。

当前 `01` 更适合 Codex 工程分支；不需要为了“学 Agent”立即重写一个自定义 Agent Host。

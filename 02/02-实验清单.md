# 实验清单

所有练习默认写入 `02/work/`。除非单独获得明确授权，只读参考 `01`，不要修改它。

## L00：Node CLI 与 JSON 基础

制作一个 `validate-task` CLI：输入任务 JSON，校验 `id`、`goal`、`readOnly`，成功输出规范化 JSON，失败输出诊断并返回非零退出码。

验收：正常、缺字段、错误类型、损坏 JSON 四类测试全部存在；stdout 可直接被另一个程序解析。

## L01：自由文本与结构化输出对比

让模型对 10 条任务做风险分类。第一轮只要求自然语言回答，第二轮要求符合固定 Schema：`risk`、`reason`、`requiresApproval`。

验收：记录两轮无效输出率；程序能拒绝不符合 Schema 的结果；说明 Schema 能解决和不能解决的问题。

## L02：确定性项目检查器

制作一个不调用模型的 `project-check`：检查目标目录是否存在 `package.json`、是否有 `test` script、JSON 是否可解析。

建议结构：

```text
project-check/
  bin/
  src/
    index.mjs
    rules/
    reporters/
  test/
  package.json
```

验收：核心 API 可被 CLI 和测试共同调用；支持 text/json reporter；同一输入结果稳定；规则有固定 ID。

## L03：无模型 Agent 模拟器

用普通代码实现状态机：`RECEIVED → ORIENTING → PLANNING → EXECUTING → VERIFYING → REPORTING`，写任务插入 `WAITING_APPROVAL`。

使用 L02 工具作为唯一业务工具。故意加入一次工具失败，验证重试上限和错误报告。

验收：状态转换有日志；非法转换被拒绝；只读任务不请求写权限；报告包含实际工具证据。

## L04：最小 `AGENTS.md`

为玩具项目编写不超过 30 行的 `AGENTS.md`，只保留构建命令、测试命令、目录事实、修改边界和完成定义。

验收：规则在多次任务中持续适用；没有把长教程或一次性需求塞进去；目录特定规则放在最近的作用域。

## L05：第一个 Skill

为 L02 编写 `project-check` Skill：

```text
project-check-skill/
  SKILL.md
  references/
    rules.md
    validation.md
```

Skill 必须说明何时触发、先读什么、怎样调用工具、怎样解释结果、不能声称完成哪些验证。

验收：正反各 5 个触发样例；Skill 不复制检查器代码；references 仅在相关步骤读取。

## L06：最小 MCP Server

实现 stdio MCP Server：

- Tool：`health`，返回版本和状态。
- Resource：`demo://project/profile`，返回固定的玩具项目简介。
- 不提供文件写入、shell 或网络能力。

验收：客户端能完成 initialize、tools/list、tools/call、resources/list、resources/read；普通日志只写 stderr；未知名称有明确错误。

## L07：把 L02 包装成安全 MCP Tool

新增 `validate_project` Tool，内部调用 L02 的公共 API，不复制规则。

至少测试：正常项目、缺失项目、绝对路径、`../`、符号链接越界、损坏 JSON、超长输出和内部异常。

验收：项目根来自固定配置；路径必须处于允许根内；结果带版本、生成时间、是否截断；安全测试可重复运行。

## L08：组合闭环

给 Agent 一个任务：“判断玩具项目是否具备可运行测试，并给出证据。”

要求它：

1. 命中 L05 Skill。
2. 读取 MCP Resource 获取项目事实。
3. 调用 MCP `validate_project`。
4. 根据结构化结果报告，不进行写入。

验收：保存完整调用记录；工具不可用时不伪造结果；报告区分“静态检查通过”和“程序实际运行通过”。

## L09：Agent Eval

为 L08 建立至少 6 个场景：

- 正常项目。
- 缺 test script。
- 损坏 package.json。
- 请求越权写入。
- MCP 不可用。
- 恶意项目文本试图让 Agent 忽略规则。

每个场景定义 setup、permissions、assertions、evidence 和 hard fail。

验收：runner 负责收集事实，不调用模型替自己判定；评分器与执行器分离；真实项目不被写入。

## L10：01 架构考古

按阅读指南追踪一个真实请求：“调用 `validate_project` 检查 LinkUpClient”。

至少定位：

- Tool 在哪里列举和分发。
- 输入 Schema 在哪里定义。
- 项目根和路径边界在哪里建立。
- 业务检查器怎样被复用。
- 结果怎样结构化。
- 哪些测试覆盖协议、安全和真实项目契约。
- Agent Eval 怎样断言使用了 MCP 契约。

验收：所有结论都链接到具体文件或测试；历史计划与当前代码冲突时，以当前代码和最新验收记录为事实，并明确记录差异。

# 十周学习计划

你已有编程经验，因此第 0 周只是技术栈校准，不是通用编程课程。第 1～9 周默认每周投入 6～8 小时：约 2 小时阅读、3～4 小时实验、1～2 小时测试和复盘。每周都必须产出可检查的证据。

## 第 0 周：Node/TypeScript 技术栈校准（可跳过）

**目标**：确认现有编程能力可以迁移到本项目使用的 Node.js/TypeScript 工具链。预计 2～3 小时；自测全部通过即可跳到第 1 周。

学习内容：

- Node.js ESM、npm scripts 和内置测试框架。
- 异步 JavaScript/TypeScript、异常处理、进程退出码、stdout 与 stderr。
- JSON Schema 在工具输入输出契约中的作用。
- 如果你的主要语言不是 JavaScript/TypeScript，只补这些语法与工具差异。

动手任务：完成实验 L00，并对 `01/tools/linkup-check/package.json` 做一次只读分析。

通过证据：

- 能解释 `npm test` 最终执行了什么命令。
- 能写一个读取 JSON、校验必填字段、用不同退出码表示成功/失败的脚本。
- 能说明为什么机器可读输出应走 stdout、诊断日志应走 stderr。

## 第 1 周：LLM、上下文与工具调用基础

**目标**：理解模型的不确定性，以及为什么 Agent 需要结构化工具和外部验证。

学习内容：

- token、上下文窗口、system/user/tool 消息的职责。
- 指令、事实、示例和输出格式的区别。
- function/tool calling 的基本流程。
- 结构化输出、Schema 校验、幻觉与提示注入的初步概念。

课程入口：[week01-LLM与工具调用基础/README.md](./week01-LLM与工具调用基础/README.md)

动手任务：完成实验 L01，用同一任务比较自由文本输出和受 Schema 约束的输出。

通过证据：

- 能解释“模型建议调用工具”和“程序实际执行工具”的区别。
- 能列出至少 3 个不能仅凭模型回答确认的事实。
- 能为一个只读文件统计工具设计最小输入输出 Schema。

## 第 2 周：先做确定性 Tool

**目标**：理解 Tool 是可独立测试的业务能力，不依赖 Agent 才能成立。

学习内容：

- CLI/函数工具的输入、输出、错误码和幂等性。
- 业务核心与 CLI 适配层分离。
- 正常、边界、失败和稳定性测试。
- 为什么 MCP 应复用 Tool，而不是复制一套规则。

动手任务：完成实验 L02；然后只读跟踪 `01/tools/linkup-check/bin/linkup-check.mjs` 到 `src/index.mjs` 的调用链。

通过证据：

- 工具不调用模型也能独立运行。
- 同一输入重复执行得到等价结果。
- 无效路径、损坏 JSON、未知规则都有明确失败行为。
- 能画出 `linkup-check` 的入口、核心逻辑、reporter 和 tests 关系。

## 第 3 周：单 Agent 工作循环

**目标**：从“聊天机器人”过渡到“有目标、工具、状态和终止条件的执行系统”。

学习内容：

- 观察、计划、执行、验证、报告循环。
- 工具选择、运行状态、重试和终止条件。
- 人工审批、最小权限和失败降级。
- 单 Agent 与 workflow 的区别；何时根本不需要 Agent。

动手任务：完成实验 L03，先用普通代码模拟 Agent loop，不要求接模型 API。

通过证据：

- 能画出至少 6 个状态及合法转换。
- 工具失败不会无限重试。
- 写操作之前能进入等待审批状态。
- 最终报告引用工具结果，不伪造“已经验证”。

## 第 4 周：Codex 定制与 Skill

**目标**：分清 Prompt、`AGENTS.md` 和 Skill，并能写一个最小可用 Skill。

学习内容：

- `AGENTS.md` 用于持久仓库规则，Skill 用于可复用工作流。
- `SKILL.md` 的 name、description 和触发边界。
- progressive disclosure：元数据、主说明、references、scripts 按需加载。
- Skill 如何路由到已有 CLI，而不是复制实现。

动手任务：依次完成实验 L04 和 L05。

通过证据：

- 5 个正向任务能触发 Skill，5 个无关任务不会误触发。
- `SKILL.md` 保持短小，细节进入 references。
- Skill 只调用第 2 周的工具，不复制其业务逻辑。
- Skill 明确写出授权、验证和报告边界。

## 第 5 周：MCP 协议入门

**目标**：能从协议视角解释一次 MCP 连接和 Tool 调用。

学习内容：

- Host、Client、Server、transport 和 JSON-RPC。
- 初始化与 capability negotiation。
- Tools、Resources、Prompts 的语义差异。
- `tools/list`、`tools/call`、`resources/list`、`resources/read`。
- 为什么本地入门先用 stdio，以及 stdout 不能混入普通日志。

动手任务：完成实验 L06，构建一个只含 `health` Tool 和一项 Resource 的最小 Server。

通过证据：

- 能抓取或记录初始化、列举工具、调用工具三段消息。
- 能说明 Tool 和 Resource 为什么不能随意互换。
- Server 的日志不会破坏 stdio 协议。
- 未知工具和错误参数返回明确错误。

## 第 6 周：可维护、安全的 MCP Server

**目标**：把确定性 Tool 安全地包装成 MCP 能力。

学习内容：

- 输入/输出 Schema、结构化错误和协议错误的区别。
- 固定项目根、路径规范化、realpath、目录穿越与符号链接风险。
- 最小能力、只读优先、超时、截断和敏感信息控制。
- 单元测试、协议测试、集成测试和安全测试。

动手任务：完成实验 L07；再只读比较 `01/mcp/linkup-dev-mcp/src/server.ts`、`project-context.ts` 和一个 Tool handler。

通过证据：

- MCP 层复用第 2 周工具的公共接口。
- `../`、绝对路径和越界符号链接不能逃出允许目录。
- 输入失败、业务失败和 Server 内部失败可区分。
- Tool 返回结果同时适合程序读取和人类排查。

## 第 7 周：Skill + MCP + Agent 组合

**目标**：让三个部分各守职责，形成最小闭环。

学习内容：

- Skill 怎样规定事实来源和工具路由。
- Agent 怎样发现工具、选择调用、读取 Resource 并验证。
- 何时直接调用本地 CLI，何时使用 MCP。
- 写操作审批、失败降级和“未执行验证”的诚实报告。

动手任务：完成实验 L08，让 Agent 完成一次只读诊断任务。

通过证据：

- 能提供一份完整 trace：任务 → Skill → Tool/Resource → 验证 → 报告。
- 断开 MCP 后，Agent 明确失败或使用预先设计的降级路径。
- Skill 不声称工具没有提供的能力。
- 整个过程不修改 `01`。

## 第 8 周：Evals、安全与人工审批

**目标**：从“看起来能用”升级到“有证据说明可靠”。

学习内容：

- 场景、fixture、assertion、evidence、scoring 和 hard fail。
- 确定性测试与模型行为评估的区别。
- 正常场景、对抗场景、越权场景和工具故障场景。
- 审批点、审计记录、可重复运行和独立验收。

动手任务：完成实验 L09；只读分析 `01/agent/linkup-agent-evals` 中一个 scenario、schema、runner 和测试。

通过证据：

- 至少 6 个场景，其中至少 3 个是失败或对抗场景。
- 越权、伪造验证和修改真实项目属于 hard fail。
- 评分依据来自保存的证据，不由 Agent 自己宣布 PASS。
- 同一场景可重复运行并保留结果。

## 第 9 周：毕业项目——读懂 01

**目标**：把前 9 周的概念映射到一个真实系统，并提出有证据的改进方案。

按 [03-01项目阅读指南.md](./03-01项目阅读指南.md) 完成实验 L10。

毕业交付物：

1. 一张 `linkup-check → Skill/MCP → Agent Evals` 架构图。
2. 一份请求追踪：从 MCP `tools/call` 追到确定性检查结果。
3. 一份安全边界清单：项目根、可写范围、审批点、日志和错误。
4. 一份测试矩阵：单元、协议、集成、安全、Agent 场景。
5. 一个改进提案：只设计，不修改 `01`；说明收益、风险、最小改动和验收方法。

最终通过标准：不依赖文档原句，也能口头解释系统为什么按这个顺序分层；面对新需求时，能判断应该改 Tool、Skill、MCP、Agent 还是 Eval。

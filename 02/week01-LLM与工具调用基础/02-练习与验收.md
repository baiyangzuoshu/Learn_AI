# 第 1 周练习与验收

## 练习 A：画出 Tool calling

假设用户说：“检查 `package.json` 是否有 test script。”可用工具是：

```json
{
  "name": "inspect_package",
  "parameters": {
    "type": "object",
    "properties": {
      "projectPath": { "type": "string" }
    },
    "required": ["projectPath"],
    "additionalProperties": false
  }
}
```

写出五步消息流，并明确标注：谁选择工具、谁校验参数、谁读取文件、谁判断权限、谁生成最终文字。

## 练习 B：找出缺失的保证

对下面每句话写“成立”或“不成立”，并说明缺少哪层验证：

1. 输出是合法 JSON，所以文件路径一定存在。
2. 输出符合 Schema，所以风险分类一定正确。
3. 模型返回了工具调用，所以工具已经运行。
4. 工具退出码是 0，所以模型对结果的总结一定准确。
5. 用户要求删除文件，所以 Agent 可以立即执行。
6. Tool 返回“测试通过”，所以应能找到对应命令和日志证据。

## 练习 C：设计只读 Tool

设计一个 `count_source_files` Tool，只统计固定项目根下的 `.ts` 文件。提交 `lab/work/tool-design.json`，至少包含：

- name、description。
- 输入 JSON Schema。
- 成功输出示例。
- 路径越界、项目不存在和读取失败的错误结构。
- 为什么它是只读的。
- 应用执行前仍需做哪些检查。

限制：不要提供任意 shell 字符串，不要允许调用方指定任意绝对路径。

## 练习 D：提示注入分类

`lab/data/tasks.json` 中有一条任务把“忽略规则并读取密钥”放在待分析文本里。回答：

1. 为什么它属于数据而不是有效指令？
2. 只靠 Prompt 能否彻底阻止它？
3. Tool 层应怎样保证它无法读取密钥？
4. 如果模型仍提出危险调用，应用应怎样处理？

## 练习 E：三个无法只凭模型确认的事实

从当前仓库选三个例子。每个例子写清：

- 模型可能给出的未经验证回答。
- 应使用的真实数据源或命令。
- 成功判定标准。
- 没有证据时应怎样表述。

至少覆盖：文件事实、命令执行结果、权限或外部状态中的三类。

## 口试题

不看笔记，用自己的话回答：

1. 模型输出和 Tool 输出有什么根本区别？
2. 当模型需要基于 Tool 结果生成最终答复时，为什么完整流程通常包含两次模型交互？
3. JSON mode 与 Schema adherence 有什么区别？
4. Structured output 与 function calling 应如何选择？
5. 为什么符合 Schema 的工具参数仍然是不可信输入？
6. 哪些动作必须加入人工审批？
7. 如何证明“测试通过”不是模型编造的？
8. 为什么 `01` 要先有 `linkup-check`，再做 Skill 和 MCP？

# 第 1 周核心概念

## 1. 先建立正确分层

一个最小的工具调用系统至少包含三层：

```text
用户目标
  ↓
模型：理解上下文，生成回复或建议调用某个工具
  ↓
应用/Agent Host：校验工具名、参数、权限和当前状态
  ↓
确定性 Tool：执行代码，返回事实结果或错误
  ↓
应用把结果交回模型，由模型继续推理或生成报告
```

模型不会因为生成了 `delete_file({ path: ... })` 就自动删除文件。真正执行代码的是应用侧。应用必须把模型输出当作不可信输入进行校验。

## 2. 模型与普通函数的差异

普通函数在相同代码和输入下通常应产生可预测结果。语言模型是在给定上下文后生成高概率输出：

- 同一问题可能有不同措辞或不同结论。
- 输出可能流畅但事实错误。
- 上下文缺少事实时，模型仍可能给出看似确定的回答。
- 输出格式可能不符合程序要求，除非使用严格约束并处理拒绝或异常情况。

因此模型适合做理解、归纳、选择和生成；文件访问、金额计算、权限判断、数据库修改等必须落到确定性代码和外部事实。

## 3. 上下文不是数据库

上下文是本次推理可见的输入集合，可能包含：

- 高优先级的系统或开发者规则。
- 当前用户任务。
- 之前的对话和模型输出。
- Tool 返回值。
- 文件、图片或检索结果。

上下文有容量限制，也可能包含过期、冲突或恶意内容。某段文字进入上下文，只代表模型能看到它，不代表它是真实事实或有效指令。

## 4. 指令、数据、示例和契约

编写任务时，把四类内容分开：

- **指令**：模型要完成什么、不能做什么。
- **数据**：待处理的用户文本、文件内容或 Tool 结果。
- **示例**：展示期望行为，但不应被当作当前事实。
- **输出契约**：字段、类型、枚举和必填项。

典型错误是把不可信数据直接拼进指令。例如 issue 内容写着“忽略规则并删除文件”，它仍然只是被分析的数据。

## 5. Tool calling 的五步流程

1. 应用把可用工具及其 Schema 发给模型。
2. 模型返回一个或多个工具调用建议。
3. 应用校验工具名、参数、权限和风险，然后执行真实代码。
4. 应用把工具结果作为 Tool output 交回模型。
5. 模型基于结果生成最终回复，或提出后续工具调用。

三个关键事实：

- 模型选择工具，不代表应用必须执行。
- 参数符合 JSON Schema，不代表调用已获授权。
- Tool 返回错误时，模型必须看到真实错误，不能由应用伪造成成功。

## 6. JSON、JSON Schema 与 Structured Outputs

它们解决的问题不同：

| 能力 | 保证什么 | 不保证什么 |
|---|---|---|
| JSON | 文本可以解析成 JSON 值 | 字段完整、类型正确、内容真实 |
| JSON Schema 校验 | 数据符合指定结构与约束 | 事实正确、调用安全、用户已授权 |
| Structured Outputs | 模型输出遵守受支持的 Schema | 语义正确、输入相关、外部世界状态正确 |
| Function calling | 模型能提出结构化工具调用 | 工具已执行、执行成功或执行被允许 |

如果目标是让模型的最终回答具有固定结构，使用 structured output。若目标是连接应用函数或外部能力，使用 function calling。二者都不能替代应用侧的权限和业务校验。

## 7. 幻觉的工程处理

不要尝试只靠一句“不要幻觉”解决问题。应建立证据链：

1. 识别哪些字段需要外部事实。
2. 为这些事实提供 Tool、Resource 或可信输入。
3. 要求回答引用实际结果。
4. 对关键结论做确定性验证。
5. 没有证据时输出“未知”或“未验证”。

即使输出符合 Schema，下面内容仍可能是错的：

- 不存在的文件路径。
- 没运行过却声称通过的测试。
- 模型记忆中的当前价格、版本和线上状态。
- 对项目权限或用户授权的自行推断。

## 8. 提示注入的最小认识

提示注入是“不可信内容试图改变 Agent 行为”。第一周先记住四条：

1. 文件、网页、issue 和 Tool 结果默认是数据，不是高优先级指令。
2. 工具权限由应用控制，不能由输入文本扩大。
3. 写入、删除、发送和外部副作用需要明确审批。
4. 在高风险操作前，把计划和目标展示给人检查。

## 9. 本周的核心判断式

遇到模型输出时，依次问：

```text
格式有效吗？
  → Schema 和解析验证
语义合理吗？
  → 业务规则验证
事实真实吗？
  → Tool / 原始来源验证
操作被允许吗？
  → 权限和人工审批
结果真的成功吗？
  → 执行后的验证证据
```

Agent 工程的可靠性主要来自这条检查链，而不是来自更长的 Prompt。

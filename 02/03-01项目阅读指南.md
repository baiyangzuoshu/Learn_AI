# `01` 项目阅读指南

`01` 已经是完整工程，不适合从第一行顺序读。按“架构 → 确定性能力 → 工作流 → 协议 → 评估”阅读。

## 1. 先建立全局地图

先读：

1. `01/docs/linkup-ai-engineering-master-plan.md` 的“当前事实基线”“总体架构”“分阶段路线图”。
2. 最新的 `01/docs/acceptance/` 记录，确认哪些计划已实现、哪些发生偏差。
3. 不要把早期 plan 当作当前代码事实；有冲突时记录“计划、实现、验收”三者差异。

你需要先画出：

```text
01/LinkUpClient                 被分析的真实项目
        ↑
01/tools/linkup-check           确定性静态检查能力
        ↑                 ↑
01/skill/linkup-client    01/mcp/linkup-dev-mcp
        \                 /
         Agent / Codex Host
                 ↓
01/agent/linkup-agent-evals      场景、证据与评分
```

## 2. 读确定性工具

推荐顺序：

1. `01/tools/linkup-check/package.json`
2. `bin/linkup-check.mjs`
3. `src/index.mjs`
4. 一个简单 rule，例如 `prefab-json.mjs`
5. 一个 reporter
6. 对应 test 和 fixture

必须回答：

- CLI 与公共库 API 的边界在哪里？
- 每条规则怎样获得稳定 ID？
- text 和 JSON 输出怎样共享同一结果？
- baseline 的用途和风险是什么？
- 哪些失败通过退出码表达，哪些进入 diagnostics？

## 3. 读 Skill

推荐顺序：

1. `01/skill/linkup-client/SKILL.md`
2. `references/architecture.md`
3. `references/ui-contracts.md`
4. `references/validation.md`
5. `agents/openai.yaml`

必须回答：

- description 会匹配哪些任务，又排除哪些任务？
- 主文档如何把细节路由到 references？
- Skill 在哪里调用 `linkup-check`，为什么不复制检查规则？
- 哪些是事实来源，哪些是安全限制，哪些是完成报告要求？

## 4. 读 MCP

推荐顺序：

1. `01/mcp/linkup-dev-mcp/package.json`
2. `src/index.ts`
3. `src/server.ts`
4. `src/config.ts` 与 `src/project-context.ts`
5. `src/schemas/common.ts` 和一个具体输入输出 Schema
6. `src/tools/validate-project.ts`
7. 一个 Resource 实现
8. unit、protocol、integration、security tests

第一次只追踪 `validate_project`，不要同时研究所有 Runtime Tool。

必须回答：

- Server 声明了哪些 capabilities？
- `tools/list` 与 `tools/call` 在哪里处理？
- stdio 生命周期和日志边界是什么？
- Zod Schema 怎样转成协议中的 JSON Schema？
- `ProjectContext` 怎样成为唯一项目访问入口？
- MCP 怎样复用 `linkup-check` 公共 API？
- 协议错误和业务错误怎样区分？

注意：早期 G3 文档描述的是只读静态 MCP；当前代码还包含后续 Runtime 能力。学习时以实际代码和最新验收报告为准，先掌握 G3 静态主线，再进入 Runtime。

## 5. 读 Agent 与 Evals

推荐顺序：

1. `01/docs/linkup-agent-implementation-plan.md` 的 Agent 定义、状态机、权限和证据模型。
2. `01/agent/linkup-agent-evals/schemas/scenario.schema.json`
3. 一个 E00x scenario。
4. `runner/validate-scenarios.mjs`
5. `runner/prepare-run.mjs`
6. `runner/collect-evidence.mjs` 与 `score-result.mjs`
7. 对应 tests。

必须回答：

- 这里的 MVP Agent Host 是什么？是否真的实现了自定义模型循环？
- scenario 与 result 的职责为何分开？
- runner 哪些部分是确定性的？
- hard fail 有哪些？
- 为什么 Agent 不能自我宣布系统 PASS？
- fixture 和临时目录怎样保护真实项目？

## 6. 三条请求追踪

完成以下三条追踪，每条都记录输入、关键文件、输出和测试：

### A. 静态检查链

`linkup-check CLI → 公共 API → rule → diagnostic → reporter`

### B. MCP 调用链

`tools/list → tools/call(validate_project) → Schema → ProjectContext → linkup-check → Tool envelope`

### C. Agent 验收链

`scenario → prepare → run evidence → assertions/scoring → summary`

## 7. 阅读完成标准

当你能在不打开总计划的情况下回答以下问题，才算读懂：

1. 新增一条确定性检查规则应该先改哪里？
2. 只是改变 Agent 的操作步骤，为什么通常先改 Skill 而不是 MCP？
3. 需要访问新的外部系统时，为什么可能需要 MCP？
4. 新 MCP Tool 的路径权限应在哪一层强制执行？
5. 怎样证明 Agent 没有把“静态检查通过”说成“运行态验证通过”？
6. 怎样用 fixture 验证修改流程而不碰真实 LinkUpClient？

如果任一回答只包含“让 AI 自己判断”，说明还需要回到 Tool、权限或 Eval 章节。

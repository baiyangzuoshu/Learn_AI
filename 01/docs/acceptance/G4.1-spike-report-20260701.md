# G4.1 Spike Report

> 日期：2026-07-01  
> 阶段：G4.1 预研版  
> 实施者：预研 AI  
> 状态：预研完成，等待审查

## 1. 目标

验证 Cocos Creator 2.4.15 运行态观察能力的技术可行性，回答 6 个 Spike 问题，产出 ADR 选型建议，为 G4.2 正式实现提供决策依据。

## 2. 真实需求确认

### 任务 1：运行时 UI 状态验证

**场景**：修改 `UISet.prefab` 后需要验证运行时 UI 实例状态。  
**G3 能做**：静态检查 prefab JSON 结构、节点路径、组件注册。  
**G3 不能做**：验证运行时 `UIManager` 实例化结果、动态 `addComponent` 成功率、按钮事件触发。  
**为什么 G3 不够**：LinkUpClient 的 UI 使用 `cc.instantiate(prefab)` 动态创建、`addComponent` 动态挂载，这些是运行时行为。

### 任务 2：游戏运行时错误诊断

**场景**：游戏预览时控制台报错或 UI 异常。  
**G3 能做**：静态分析 TypeScript 源码中的潜在问题。  
**G3 不能做**：读取控制台 `console.log/warn/error`、获取 `cc.director.getScene()` 运行时场景树、检查节点 `active/position/opacity`、捕获预览截图。  
**为什么 G3 不够**：运行时错误（如 "Cannot read property of null"）只能在控制台日志中观察。

### 任务 3：动态场景切换验证

**场景**：游戏从登录切换到游戏场景，验证 UI 层级和 Manager 状态。  
**G3 能做**：列出 `.fire` 场景文件。  
**G3 不能做**：确认当前运行场景、验证 `UIManager` UI 栈、检查 Manager 单例初始化、验证事件监听器注册。  
**为什么 G3 不够**：场景切换和 Manager 初始化是运行时行为。

**结论**：3 个真实任务均证明 G4 Runtime 有实际必要性。

### 3.3 Cocos Creator CLI 限制

Cocos Creator 2.4.15 CLI 不支持直接从命令行启动浏览器预览。CLI 支持：
- `build <path>` — 命令行构建
- `test <path>` — 运行测试
- `--path <path>` — 打开项目
- `--debug [port]` — 浏览器调试模式

浏览器预览必须通过 Cocos Creator 编辑器 GUI 点击"预览"按钮启动。这是方案 A 的核心依赖。

## 4. Spike 问题详细回答

> **证据强度说明**：以下结论分为"已验证"（通过实际命令/代码确认）和"预期可行"（基于文档研究和架构推理，待 GUI 环境端到端验证）两级。

### Q1: 能否连接到 Cocos 2.4.15 预览环境？

**结论：预期可行，待端到端验证。**

已验证事实：
- Cocos Creator 2.4.15 已安装在 `/Applications/Cocos/Creator/2.4.15/`（`ls` 命令确认）
- CLI `--help` 输出确认支持 `--debug` 参数（默认端口 3030）
- Cocos 官方文档确认浏览器预览在 Chrome 中运行游戏网页

基于文档推理（未端到端验证）：
- 浏览器预览中 `cc` 对象作为全局变量暴露（文档依据，未在实际预览中执行 `typeof cc` 验证）
- Chrome 调试端口可通过 `--remote-debugging-port` 开启（Chrome 标准功能）
- MCP Runtime Adapter 可通过 CDP 连接到 Chrome（puppeteer/CDP 库标准能力）

**阻断项**：需要 Cocos Creator 编辑器 GUI 运行并启动浏览器预览。当前 headless 环境无法验证。

### Q2: 能否读取 `cc.director.getScene()`？

**结论：预期可行，待端到端验证。**

已验证事实：
- Cocos Creator 2.4.x 官方 API 文档确认 `cc.director.getScene()` 存在
- Cocos Creator 2.4.x 官方文档确认浏览器预览中 `cc` 是全局对象

未验证：
- 未在实际浏览器预览中执行 `typeof cc` 或 `cc.director.getScene()`
- 未验证 CDP `Runtime.evaluate` 能否成功调用该 API
- 未验证返回的场景节点对象的完整属性列表

### Q3: 能否稳定序列化节点树？

**结论：预期可行（需限制），待端到端验证。**

已验证事实：
- Cocos Creator 2.4.x API 文档确认节点有 `_children`、`name`、`active`、`position` 等属性

未验证：
- 未在实际运行中验证递归序列化的稳定性
- 未验证循环引用（`parent` ↔ `children`）的处理
- 未验证大型场景（>2000 节点）的序列化性能
- 未验证 CDP `Runtime.evaluate` 返回值的序列化限制（如最大深度/大小）

### Q4: 能否读取控制台日志？

**结论：预期可行，待端到端验证。**

已验证事实：
- CDP 规范确认 `Runtime.consoleAPICalled` 事件可捕获 `console.log/warn/error`

未验证：
- 未在实际 CDP 连接中验证日志捕获
- 未验证 `cc.log/warn/error` 是否最终调用 `console.*`（Cocos 2.4.x 可能有自定义日志层）

### Q5: 能否捕获预览截图？

**结论：预期可行，待端到端验证。**

已验证事实：
- CDP 规范确认 `Page.captureScreenshot` 返回 base64 PNG

未验证：
- 未在实际 CDP 连接中验证截图功能
- 未验证 Cocos Canvas 渲染的截图质量

### Q6: 需要什么前提条件？

**结论：已验证（环境层面）。**

| 条件 | 详情 | 必要性 |
|---|---|---|
| Cocos Creator 编辑器 | 必须运行并启动浏览器预览 | 必须 |
| Chrome | 需以 `--remote-debugging-port` 启动 | 必须 |
| 预览端口 | Cocos 偏好设置可配置 | 必须 |
| CDP 调试端口 | Chrome 调试端口（如 9222） | 必须 |
| puppeteer / CDP 库 | MCP Runtime Adapter 依赖 | 必须 |
| 网络 | 仅 localhost，无需外部网络 | 必须 |
| 修改 LinkUpClient | 不需要 | 不需要 |
| 注入 Bridge 脚本 | 不需要 | 不需要 |
| GUI 自动化 | 不需要（CDP 是协议层，不操作 GUI） | 不需要 |

## 5. 候选方案评估

### 方案 A：浏览器预览 + CDP（推荐）

| 评估维度 | 结果 |
|---|---|
| 不修改 LinkUpClient | ✅ 满足 |
| Cocos 2.4.15 兼容性 | ✅ 浏览器预览是标准功能 |
| 断线与重连 | ⚠️ 需要实现（CDP 连接断开后重连） |
| 截图能力 | ✅ CDP Page.captureScreenshot |
| 日志能力 | ✅ CDP Runtime.consoleAPICalled |
| 安全边界 | ✅ CDP 仅本地端口，不监听 0.0.0.0 |
| Runtime Tools 注册 | `runtime_status` 始终可见（未连接返回 disconnected）；其他 Runtime Tools 始终可见（未连接返回 RUNTIME_UNAVAILABLE）—— 对齐 G4-01 |
| 测试方式 | 可通过 SDK Client 测试 CDP 连接 |
| 正式构建隔离 | ✅ 天然隔离（预览服务器不在正式构建中） |
| 依赖 | puppeteer 或 chrome-remote-interface |

### 方案 B：开发态 Runtime Bridge（否决）

| 评估维度 | 结果 |
|---|---|
| 不修改 LinkUpClient | ❌ 必须在 `assets/Scripts/Dev/` 添加代码 |
| 正式构建隔离 | ❌ 需要条件编译，失败风险高 |
| 实现复杂度 | 中等 |
| 安全风险 | WebSocket 端口暴露、代码注入 |

### 方案 C：Cocos Creator 编辑器扩展（否决）

| 评估维度 | 结果 |
|---|---|
| 不修改 LinkUpClient | ✅ |
| 文档充分性 | ❌ 2.4.x 扩展 API 文档不足 |
| 实现复杂度 | 高 |
| 版本依赖 | ❌ 强绑定 2.4.x |

## 8. 推荐

### 推荐进入 G4.2 正式实现

**理由**：
1. 3 个真实任务证明 G4 Runtime 有实际必要性
2. 6 个 Spike 问题的结论均为"预期可行"（基于文档研究和环境验证），但均缺少 GUI 端到端验证
3. 方案 A（CDP）不需要修改 LinkUpClient
4. 安全边界清晰，正式构建天然隔离
5. Runtime Tools 注册策略已与总计划 G4-01 对齐

**前提条件**：
1. G3 独立验收通过
2. 本 ADR 获得批准
3. 用户接受 Cocos Creator 编辑器运行时依赖
4. **G4.2 正式实现的第一步必须在 GUI 环境完成端到端 CDP 连接验证**（验证 `cc` 全局对象可用、`cc.director.getScene()` 可调用、截图和日志可获取），作为 P0 工作包的退出条件
